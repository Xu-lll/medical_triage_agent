# 智能医疗分诊与问诊智能体面试指南

## 1. 项目一句话介绍

这个项目面向线上问诊和预分诊场景，基于 FastAPI、SQLite 医学知识库、Chroma 向量数据库、LangChain、Ollama 本地大模型和 Web 前端，实现了一个可连续对话的医疗问诊智能体。用户可以像和聊天机器人一样描述症状、补充病情、询问挂什么科或用药注意事项；系统会结合历史上下文、RAG 检索结果、急症红旗规则和 LLM 生成能力，给出阶段性分诊建议、进一步追问和安全提醒。

项目定位不是替代医生诊断，而是做“就医前的信息整理和分诊辅助”。

## 2. 当前核心功能

- 多轮问诊：支持用户连续补充“从昨天开始”“没有其他症状”“该挂什么科”等上下文信息。
- 双模式对话：支持“纯 LLM 对话”和“RAG 知识库增强”两种模式。
- LLM 主导回复：最终回答、阶段性建议和追问由 LLM 生成，规则主要负责安全兜底和证据整理。
- RAG 检索增强：使用 SQLite 存储结构化医学知识，使用 Chroma 做向量检索，并结合关键词混合排序。
- 急症红旗筛查：对胸痛、呼吸困难、剧烈腹痛、卒中信号等做安全优先判断。
- 科室推荐：根据检索文档相关性、风险等级和关键词命中情况生成 Top 科室候选。
- 意图识别：区分症状描述、病情补充、挂号科室咨询、用药咨询和普通聊天。
- 响应耗时记录：每次 `/api/chat` 请求会返回响应耗时，并追加写入 `data/response_times.jsonl`。
- 前端演示：Web 页面支持连续聊天、清空会话、切换问诊模式、查看响应时间和检索依据。
- 本地运行：通过 Ollama 本地模型运行，适合毕业设计、答辩和面试演示。

## 3. 技术架构

```text
用户输入
  ↓
FastAPI /api/chat
  ↓
会话管理 session_id
  ↓
MedicalTriageAgent
  ├─ 意图识别
  ├─ 急症红旗筛查
  ├─ 历史上下文整合
  ├─ RAG 检索，纯 LLM 模式可跳过
  ├─ 科室候选排序
  ├─ LLM 生成本轮回复
  └─ 规则兜底与输出清洗
  ↓
Web 前端展示答案、耗时、证据和轨迹
```

项目主要模块：

- `app/agent.py`：核心问诊 Agent，负责多轮上下文、意图识别、检索、科室排序和 LLM 生成。
- `app/api.py`：FastAPI 接口，负责会话隔离、请求响应、耗时记录和前端页面服务。
- `app/retriever.py`：混合检索逻辑，融合关键词命中和向量召回。
- `app/vector_store.py`：Chroma 向量库构建和查询。
- `data/medical_seed.json`：医学知识库种子数据。
- `scripts/init_db.py`：初始化 SQLite 知识库。
- `scripts/build_vector_db.py`：重建 Chroma 向量库。
- `tests/test_agent.py`：核心行为测试。

## 4. 为什么要做双模式

一开始系统主要是“输入症状，然后输出一次完整分诊报告”。这个形态更像规则系统，不像真正的问诊智能体。后来增加了两个模式：

1. 纯 LLM 对话  
   不调用本地知识库，主要验证大模型自然对话能力。适合用户问一些不完全是症状的问题，比如“我该注意什么”“这严重吗”“我该挂什么科”。

2. RAG 知识库增强  
   先检索本地医学知识库，再让 LLM 结合证据生成回答。适合面试展示可解释性，因为可以说明建议来自哪些医学条目。

这样设计的好处是，既能体现大模型对话能力，又能展示 RAG 在医疗垂直场景中的可控性和可解释性。

## 5. LLM 在项目中做了什么

当前版本不是简单把 LLM 用来“润色模板”，而是让 LLM 主导本轮问诊回复：

- 根据用户当前输入和历史对话判断应该先回答什么。
- 在信息不足时生成更贴合当前病情的追问。
- 在用户询问科室时直接给出科室建议，而不是继续重复追问。
- 在用户询问用药时给出安全边界，例如不提供处方剂量，提示过敏史、基础病、儿童孕妇等风险。
- 在 RAG 模式下结合检索知识、科室候选和红旗信号生成答案。
- 在纯 LLM 模式下不展示本地知识库证据，让对话更像通用问诊助手。

规则系统仍然保留，主要用于：

- 急症红旗兜底。
- 检索证据过滤。
- 科室候选排序。
- LLM 调用失败时提供备用回答。
- 清理 LLM 输出中的内部提示词泄露。

## 6. RAG 为什么有必要

如果完全依赖通用大模型，医疗场景会有几个风险：

- 可能编造不存在的依据。
- 对具体科室分诊的稳定性不够。
- 对急症和非急症边界不够可控。
- 每次回答风格和结论可能波动较大。

RAG 的价值是把本地医学知识库作为上下文提供给模型，让模型基于已整理的疾病、症状、科室和风险规则生成回答。

在这个项目里，RAG 的作用主要体现在：

- 可解释：前端可以展示检索到的医学条目和来源。
- 可更新：补充 `data/medical_seed.json` 后重建数据库和向量库即可扩展能力。
- 可控：可以过滤低相关文档，减少“随便召回一个不相关疾病”的问题。
- 易演示：本地 SQLite + Chroma 不依赖外部云服务。

## 7. 本次完善过程总结

这次完善主要围绕“让系统更像问诊智能体，而不是固定分诊模板”展开。

### 7.1 问题一：不管输入什么都容易建议急诊

早期急症判断比较粗糙，只要命中“胸痛、头痛、腹痛”等词，就容易给出急诊建议。这样会导致普通感冒、轻微胸口不适、普通腹痛都被过度分诊。

解决方式：

- 把急症判断从“单词命中”改成“明确红旗词 + 症状组合”。
- 明确红旗词包括意识障碍、说话不清、一侧无力、大出血、血氧下降等。
- 泛化症状需要组合判断，例如胸痛伴大汗、放射痛、气短、晕厥才优先急诊。
- 增加测试用例，确保普通感冒、单纯胸口不适不会自动进入急诊。

面试可以这样讲：

> 医疗分诊宁可保守，但不能把所有症状都推急诊。我把急症识别拆成明确红旗和组合红旗两层，既保留高风险场景的安全性，也减少普通症状的误报。

### 7.2 问题二：每一轮回答格式都一样

用户反馈系统每次都输出类似固定模板，包括“患者本轮描述、检索知识、科室候选、追问问题”等内容。这更像调试信息，不像真实聊天机器人。

解决方式：

- 把内部检索结果、规则兜底和提示词上下文从用户可见回答中剥离。
- 让 LLM 直接输出面向患者的自然语言回复。
- 前端只展示最终答案，调试信息只在需要时通过 trace 或证据区域查看。
- 增加输出清洗逻辑，避免 `internal_context`、`final_answer` 等内部字段泄露到前端。

面试可以这样讲：

> 早期版本为了方便调试，把中间过程也输出给用户了。后来我把系统分成“内部推理上下文”和“用户可见回复”两层，既保留可调试性，又让前端体验更自然。

### 7.3 问题三：用户补充病情后，系统重复问同样的问题

例如用户先输入“我头疼发热怎么办”，系统追问病程和伴随症状；用户回答“从昨天开始，加重，没有其他症状”，系统仍然重复问“从什么时候开始，有没有其他症状”。

原因：

- 每轮输入被当作独立问题处理。
- 追问问题是固定模板，不能识别用户已经回答过的信息。
- 检索输入没有充分融合历史上下文。

解决方式：

- 为每个 `session_id` 保存独立 Agent 实例。
- 在 Agent 内保留最近对话和简易长期摘要。
- 从历史对话中抽取已知事实，例如病程、是否加重、主要症状、是否有其他伴随症状。
- LLM 生成追问时把“已知信息”放入上下文，要求不要重复询问已经回答过的问题。
- 增加测试 `test_no_other_symptoms_does_not_repeat_other_symptom_question`。

面试可以这样讲：

> 多轮问诊的关键不是把聊天记录简单拼接，而是要把已知事实结构化地提供给模型。否则模型虽然看到了历史，但仍然可能按模板重复提问。

### 7.4 问题四：用户问“该挂什么科”时没有直接回答

用户输入“我口腔起了很多水泡，有点疼，该去什么诊室”，系统只提示保持口腔清洁，没有明确给出科室。

原因：

- 知识库缺少口腔水泡、口腔黏膜、口唇疱疹等相关条目。
- “该去什么诊室”没有被足够明确地识别为科室咨询意图。
- 向量检索在知识覆盖不足时只能返回低相关结果，导致科室排序不稳定。

解决方式：

- 在 `data/medical_seed.json` 中新增口腔相关知识：
  - 口腔溃疡/口腔黏膜破损
  - 口唇疱疹
  - 口腔水泡危险信号
- 扩展通用医学词和查询扩展逻辑，例如“口腔水泡”“嘴里起泡”“口腔溃疡”。
- 针对科室咨询意图增加直接回答逻辑。
- 重建 SQLite 和 Chroma 向量库，当前向量库文档数量为 53。
- 增加测试 `test_oral_blisters_route_to_stomatology`。

面试可以这样讲：

> 这个问题暴露了 RAG 系统的一个典型短板：不是模型不会答，而是知识覆盖和召回策略没有覆盖到这个表达。我的处理不是只改提示词，而是同时补知识、补关键词、补意图识别和补测试。

### 7.5 问题五：RAG 检索到不相关知识

用户输入“便血、腹痛好几天”，早期可能检索到胸痛、卒中、流感等 0 分或低相关文档。这会影响 LLM 上下文，导致回答跑偏。

解决方式：

- 增加低相关文档过滤逻辑。
- 对“无直接关键词命中”的文档降低可见性。
- 当检索证据不可靠时，提示 LLM 不要硬套知识库。
- 在 RAG 模式下保留“检索知识可能为空或不完全相关”的安全边界。

面试可以这样讲：

> RAG 不是检索越多越好。医疗场景下，低相关文档进入上下文反而会误导模型，所以我增加了证据过滤和可靠性判断。

### 7.6 问题六：LLM 输出了内部提示词

有一次用户看到回答里出现“请直接输出给患者看的本轮问诊回复，不要包含 internal_context 或任何内部字段标题”。这说明 LLM 把系统提示的一部分当成了最终答案输出。

解决方式：

- 在提示词中用明确标签区分内部上下文和最终回答。
- 增加 `_strip_internal_prompt_echo` 清洗函数，过滤内部字段标题和提示词片段。
- 强制最终返回只保留患者可读内容。
- 如果 LLM 输出为空或明显异常，则走规则兜底回答。

面试可以这样讲：

> 这是 LLM 应用里常见的 prompt leakage 问题。我没有只依赖提示词约束，而是在输出层又做了一次后处理和兜底，保证用户看不到内部控制字段。

## 8. 混合检索怎么做

项目没有只用向量检索，而是采用“向量召回 + 关键词命中”的混合检索。

原因：

- 向量检索擅长语义相似，例如“喘不上气”和“呼吸困难”。
- 关键词检索擅长精确症状，例如“血便、尿痛、口腔水泡”。
- 医疗红旗信号必须尽量精确命中，不能只依赖语义相似。

综合排序大致考虑：

- 向量相似度。
- 关键词命中数量。
- 文档风险等级。
- 文档是否与当前症状直接相关。
- 是否存在科室咨询、用药咨询等特殊意图。

## 9. 科室推荐怎么来

检索返回的是医学知识条目，但用户真正关心的是“该去哪个科”。系统会把相关文档聚合到科室维度：

```text
检索文档 → 文档分数 → 科室聚合 → Top 科室候选
```

例如：

- 发热、咳嗽、咽痛：可能聚合到呼吸内科、发热门诊、全科。
- 尿频、尿急、尿痛、血尿：可能聚合到泌尿外科。
- 口腔水泡、口腔疼痛：优先聚合到口腔科，必要时提示皮肤科。

如果用户明确问“挂什么科”“该去什么诊室”，系统会优先直接回答科室，而不是继续泛泛追问。

## 10. 为什么不用完全规则系统

完全规则系统的问题是：

- 对自然语言表达不够灵活。
- 追问容易固定，用户体验像填表。
- 用户问一些非标准问题时容易无响应。
- 维护成本高，每个场景都要写模板。

所以当前版本改成 LLM 主导生成，规则和知识库负责边界控制：

```text
LLM 负责自然语言、多轮对话、动态追问
RAG 负责医学依据和可解释性
规则负责急症安全和兜底稳定性
```

## 11. 为什么不用完全 LLM

完全 LLM 的问题是：

- 医疗建议可能不稳定。
- 科室推荐可能前后不一致。
- 很难解释依据来自哪里。
- 不能保证急症红旗一定被正确处理。

因此项目采用折中方案：LLM 主导表达和问诊，RAG 与规则负责安全边界。

面试可以用一句话概括：

> 这个项目不是把大模型当万能医生，而是把它放在一个受约束的问诊流程里，让它负责对话和推理表达，让知识库和规则负责证据与安全。

## 12. 数据库设计

SQLite 中主要是一张 `knowledge` 表：

```text
id          知识条目 ID
category    类型：疾病、药品、分诊规则等
title       标题
department  推荐科室
severity    风险等级
keywords    关键词
content     医学知识内容
source      来源链接
```

这样设计的优点：

- 结构简单，适合小型医学知识库。
- 每条知识天然带有科室、风险等级和来源。
- 可以直接用于检索、排序、展示和评测。
- 后续可以扩展 ICD 编码、适用人群、禁忌症、年龄限制等字段。

面试可以这样讲：

## 15. 面试高频问答

### 问题 1：这个项目解决什么问题？

回答：

它解决的是线上问诊场景中用户不知道挂什么科、症状描述不规范、初步健康建议缺少依据的问题。系统通过多轮对话收集症状信息，再结合 RAG 知识库、急症红旗规则和 LLM 生成能力，给出阶段性分诊建议和安全提醒。

### 问题 2：项目里大模型到底在哪里用到了？

回答：

大模型主要用于多轮对话生成，包括理解用户补充信息、生成追问、回答挂号科室问题、给出阶段性建议和用药安全提醒。RAG 模式下，大模型还会结合检索到的医学知识生成回答；纯 LLM 模式下则不调用本地知识库，用来验证自然对话能力。

### 问题 3：为什么要保留规则？

回答：

医疗场景不能完全依赖大模型。规则主要处理急症红旗和安全兜底，比如意识障碍、说话不清、胸痛伴大汗等高风险场景必须优先处理。LLM 负责自然语言和动态问诊，规则负责安全底线。

### 问题 4：为什么做 RAG 和纯 LLM 两个模式？

回答：

纯 LLM 模式更像通用聊天机器人，适合测试自然对话；RAG 模式能结合本地医学知识库，输出更可解释、更稳定的分诊建议。两个模式可以对比展示知识库增强对医疗垂直场景的价值。

### 问题 5：遇到重复回答怎么解决？

回答：

我把每轮对话从独立请求改成带会话上下文的请求。后端通过 `session_id` 隔离不同会话，Agent 内部保存近期对话，并抽取已知事实。生成追问时会把已知信息交给 LLM，避免重复问已经回答过的问题。

### 问题 6：遇到知识库没有覆盖怎么办？

回答：

这次口腔水泡就是一个例子。我先确认系统没有稳定给出口腔科，然后补充口腔溃疡、口唇疱疹、口腔水泡危险信号等知识条目，再扩展关键词和查询扩展逻辑，最后重建 SQLite 和 Chroma，并添加单元测试防止回归。

### 问题 7：如何减少医疗幻觉？

回答：

主要靠四层控制：第一，RAG 提供可追溯知识依据；第二，低相关检索结果会被过滤；第三，急症和科室排序有规则兜底；第四，输出层限制处方剂量和内部提示词泄露。如果 LLM 输出异常，会回退到规则回答。

### 问题 8：这个项目有什么不足？

回答：

知识库规模仍然较小，评测集也是自建小样本，不能代表真实临床准确率。当前模型和 embedding 也不是专门医学模型。后续可以扩展真实问诊数据、引入医生标注、增加 BM25 + rerank、构建更细粒度的疾病和科室体系。

### 问题 9：如果继续优化，你会怎么做？

回答：

我会从四个方向继续优化：扩大医学知识库和评测集；增加 BM25、向量检索和 reranker 的组合检索；把已知症状结构化成问诊表单状态；增加不同人群的安全策略，比如儿童、孕妇、老人和慢病患者。

## 16. 可以背的项目介绍

我做的是一个面向线上问诊场景的智能医疗分诊与问诊系统。用户输入自然语言症状后，系统会先识别是否存在急症红旗，然后根据会话上下文判断用户是在描述症状、补充病情、询问科室还是咨询用药。RAG 模式下，系统会通过 SQLite 医学知识库和 Chroma 向量数据库召回相关疾病、症状和分诊规则，再结合关键词命中做混合排序，生成候选科室和医学依据。

在大模型部分，我使用 Ollama 本地部署 DeepSeek 模型，并通过 LangChain 接入问诊流程。当前版本不是简单模板回复，而是由 LLM 主导生成本轮患者可见回答，包括阶段性建议、进一步追问和安全提醒。规则系统主要负责急症兜底、检索过滤、科室候选和 LLM 异常时的备用回答。项目还提供了纯 LLM 和 RAG 知识库增强两种模式，可以对比展示自然对话能力和知识库增强后的可解释性。

这次迭代中，我重点解决了几个实际问题：普通症状被过度推荐急诊、每轮回答格式固定、用户补充信息后重复追问、知识库缺少口腔水泡导致无法推荐口腔科，以及 LLM 偶尔输出内部提示词。最终通过调整急症组合规则、增加会话记忆和已知事实抽取、补充医学知识库、过滤低相关检索结果、增加输出清洗和单元测试，让系统更接近真实的多轮问诊智能体。

## 18. 更新并覆盖到 GitHub 的命令

你之前已经上传过一个版本。后续本地修改完成后，正常情况不需要删除仓库，也不需要强制覆盖。只要在同一个分支提交并 push，GitHub 上的项目内容就会更新为最新版本。

查看当前改动：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' status
```

添加全部改动：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' add .
```

提交：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' commit -m "Improve LLM triage conversation and knowledge base"
```

推送到 GitHub：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' push origin main
```

如果你还遇到 GitHub 连接被重置，并且本机代理 HTTP(S) 端口是 `7899`，可以先设置代理：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' config --global http.proxy http://127.0.0.1:7899
& 'C:\Program Files\Git\cmd\git.exe' config --global https.proxy http://127.0.0.1:7899
```

然后再执行：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' push origin main
```

如果 GitHub 远程分支和本地分支发生冲突，优先使用：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' pull --rebase origin main
& 'C:\Program Files\Git\cmd\git.exe' push origin main
```

只有在你明确确认“远程仓库内容不要了，以本地为准”时，才使用强制覆盖：

```powershell
& 'C:\Program Files\Git\cmd\git.exe' push --force-with-lease origin main
```

`--force-with-lease` 比普通 `--force` 更安全，它会在远程分支没有被别人更新时才覆盖。
