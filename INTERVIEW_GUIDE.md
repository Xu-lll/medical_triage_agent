# 智能医疗分诊与建议智能体面试总结文档

## 1. 项目一句话介绍

这个项目面向线上问诊场景，构建了一个基于 RAG 和 Agent 协作的智能医疗分诊系统。用户输入自然语言症状描述后，系统会先从本地医学知识库中检索相关疾病、症状、药品和分诊规则，再结合急症红旗规则、科室排序策略和本地 LLM，输出推荐科室、就诊优先级、医学依据和初步建议。

最新完善点：

- 急症筛查从“单词命中即急诊”调整为“明确红旗词 + 症状组合”策略，避免普通感冒、反酸烧心、轻微胸口不适等描述被过度推荐急诊。
- 对胸痛、呼吸困难、腹痛、头痛等高风险方向保留组合红旗判断，例如胸痛伴大汗、放射痛、气短或晕厥仍会进入急症优先。
- `/api/chat` 返回 `response_time_ms`，核心 Agent 返回 `agent_response_time_ms`；所有 HTTP 响应耗时会追加记录到 `data/response_times.jsonl`。
- 前端页面在“分诊建议”区域直接展示本次响应时间，便于演示和性能观察。

项目核心技术包括：

- SQLite 医学结构化知识库
- Chroma 本地向量数据库
- Ollama 本地部署 `deepseek-r1:7b`
- LangChain 0.3
- 关键词 + 向量混合检索
- LangChain Tool / AgentExecutor 的 ReAct 工具调用流程
- FastAPI + Web 前端
- 标注问诊集评测 Top-1 准确率和 Top-3 召回率

## 2. 项目整体流程

用户输入症状描述后，系统执行以下流程：

1. 急症红旗识别  
   检查是否出现意识障碍、说话不清、一侧无力等明确红旗词；对胸痛、呼吸困难、腹痛、头痛等泛化症状，再结合大汗、放射痛、血氧下降、神经功能异常等伴随信号判断是否急症优先。

2. 医学知识检索  
   使用 Chroma 向量数据库进行语义召回，同时结合关键词命中分数进行混合排序。

3. 科室排序  
   根据检索文档的相关性分数、风险等级、关键词命中数量，对候选科室进行聚合排序。

4. Agent 工具调用  
   AgentExecutor 可以调用 `emergency_screen`、`medical_search`、`department_rank`、`followup_questions`、`compose_answer` 等工具。

5. 生成最终回答  
   系统先生成确定性的分诊结果，再让 LLM 基于检索证据生成补充说明。这样既使用了大模型，又降低了医疗场景下的幻觉风险。

## 3. 为什么使用 RAG

如果直接把用户问题交给通用大模型回答，模型可能会出现三个问题：

- 医学知识不一定最新或准确。
- 容易生成没有来源依据的回答。
- 对具体分诊规则和科室推荐缺少可控性。

RAG 的作用是把外部医学知识库作为上下文提供给模型。模型不是凭空回答，而是基于检索到的疾病、症状、药品、分诊规则生成建议。

在这个项目中，RAG 的价值主要体现在：

- 可解释：每条建议都能回溯到知识库来源。
- 可控：可以限定模型只基于检索证据回答。
- 可更新：扩充或修改医学知识库后，重建向量库即可更新系统能力。
- 更适合垂直领域：相比通用模型，医学分诊需要明确规则和风险边界。

## 4. 向量数据库的原理

向量数据库的核心思想是把文本转成向量。语义相近的文本，在向量空间中的距离也更近。

本项目中，每条医学知识会被组织成类似这样的文本：

```text
标题：尿路感染
推荐科室：泌尿外科
风险等级：medium
关键词：尿频、尿急、尿痛、血尿
医学知识：尿频、尿急、尿痛提示尿路感染可能...
来源：https://medlineplus.gov/urinarytractinfections.html
```

然后使用 Ollama 中的 `deepseek-r1:7b` embedding 接口生成向量，并存入 Chroma。

用户输入症状时，也会被转成向量。系统在 Chroma 中查找距离最近的医学知识，得到语义相似的候选文档。

## 5. 为什么使用 Chroma

选择 Chroma 的原因：

- 本地运行，不依赖云服务，适合毕业设计和面试演示。
- 和 LangChain 集成简单。
- 支持持久化，向量库可以保存在 `data/chroma_medical`。
- 小规模知识库场景下性能足够。
- 部署成本低，不需要额外安装 Milvus、Elasticsearch 等复杂服务。

如果面试官追问为什么不用 FAISS，可以这样回答：

FAISS 更偏底层向量检索库，性能强，但缺少完整的文档元数据管理和服务化能力。这个项目知识规模不大，更重视 LangChain 集成、持久化和开发效率，所以选择 Chroma。如果知识库规模扩大到百万级，可以考虑 FAISS、Milvus 或 Elasticsearch 混合检索方案。

## 6. 为什么使用 Ollama 和 DeepSeek R1

项目使用 Ollama 调用本地 `deepseek-r1:7b`，主要原因是：

- 本地部署，保护问诊隐私。
- 不依赖外部 API，演示更稳定。
- 成本低，适合学生项目。
- DeepSeek R1 有较强的推理和中文表达能力。
- 可以通过 LangChain 统一接入 LLM 和 embedding。

需要注意的是，医疗场景不能完全信任 LLM。因此项目没有让 LLM 直接决定最终分诊结果，而是采用“检索和规则先确定结论，LLM 只做补充说明”的方式。

## 7. 混合检索是怎么做的

项目没有只用向量检索，而是使用“向量分数 + 关键词分数”的混合检索。

原因是：

- 向量检索擅长语义相似，比如“喘不上气”和“呼吸困难”。
- 关键词检索擅长精确匹配，比如“尿频、尿急、尿痛、血尿”。
- 医疗分诊中很多红旗症状必须精确命中，不能只靠语义相似。

项目中的综合分数大致由两部分组成：

```text
综合分数 = 0.72 * 向量相似度 + 0.28 * 关键词命中分数 + 关键词数量奖励
```

向量相似度来自 Chroma 返回的距离，关键词命中分数来自知识库中 `keywords` 字段与用户输入的匹配情况。

## 8. 科室排序是怎么来的

检索返回的是医学知识文档，但用户真正需要的是科室推荐。所以系统会把文档分数聚合到科室上。

例如用户输入：

```text
尿频尿急尿痛，还有一点血尿
```

检索结果可能包括：

- 尿路感染：泌尿外科
- 肾结石风险：泌尿外科
- 阴道炎：妇科

系统会把同一科室下的高相关文档分数聚合，得到 Top-3 科室。

聚合时考虑：

- 文档检索相关性
- 风险等级权重
- 关键词命中数量
- 过滤低相关文档，避免多个弱相关文档把某个科室错误堆高

这也是为什么项目中 `腰痛` 评测样本最初被泌尿外科抢分，后来通过过滤低相关文档和优化关键词后修正为骨科。

## 9. ReAct 和 AgentExecutor 是什么

ReAct 是 Reasoning + Acting 的缩写，也就是“推理 + 行动”。

它的基本流程是：

```text
Thought: 我需要先判断是否急症
Action: emergency_screen
Observation: 命中胸痛、呼吸困难
Thought: 需要检索医学知识
Action: medical_search
Observation: 返回胸痛急诊规则、肺炎风险等
Thought: 需要排序科室
Action: department_rank
Observation: 急诊科第一，呼吸内科第二
Final Answer: 给出分诊建议
```

项目中定义了 5 个工具：

- `emergency_screen`：识别急症红旗词。
- `medical_search`：调用 Chroma + 关键词混合检索。
- `department_rank`：根据检索结果排序科室。
- `followup_questions`：生成追问问题。
- `compose_answer`：生成最终分诊回答。

面试中可以强调：  
AgentExecutor 的价值不是让大模型随便回答，而是让大模型根据任务需要选择工具，把复杂流程拆成可观察、可调试、可扩展的工具调用链。

## 10. 为什么最终回答不完全交给 LLM

医疗场景对安全性要求高。如果完全让 LLM 生成结果，可能会：

- 编造疾病或症状。
- 修改急症等级。
- 给出不合适用药建议。
- 忽略高风险红旗信号。

所以项目采用了两层策略：

1. 确定性分诊层  
   由检索结果、规则和科室排序算法生成核心结论。

2. LLM 补充说明层  
   LLM 只能基于检索证据补充解释，不能改变推荐科室和风险等级。

这样可以兼顾大模型的自然语言表达能力和医疗系统的安全可控性。

## 11. 数据库是怎么设计的

SQLite 中主要有一张 `knowledge` 表：

```text
id           知识条目 ID
category     类型：疾病、药品、分诊规则
title        标题
department   推荐科室
severity     风险等级
keywords     关键词
content      医学知识内容
source       来源链接
```

这样设计的好处是：

- 结构简单，适合小型知识库。
- 每条知识天然带有科室、风险等级和来源。
- 可以直接用于检索、分诊排序和引用展示。
- 后续可以扩展疾病 ICD 编码、适用人群、禁忌症等字段。

## 12. 结果指标是怎么来的

项目内置了 20 条人工标注问诊样本，保存在：

```text
data/eval_cases.json
```

每条样本包含：

```json
{
  "query": "尿频尿急尿痛，今天还有一点血尿",
  "department": "泌尿外科"
}
```

评测流程：

1. 对每条问诊调用检索器。
2. 得到 Top-3 推荐科室。
3. 判断 Top-1 是否等于标注科室。
4. 判断标注科室是否出现在 Top-3。
5. 计算平均准确率和召回率。

当前结果：

```text
评测样本数：20
Top-1 分诊准确率：1.00
Top-3 分诊召回率：1.00
知识库文档数：40
```

面试时要注意说明：  
这是小规模自建评测集上的结果，主要用于证明系统流程和工程可行性，不代表真实临床场景准确率。真实上线需要更大规模、多医生标注和临床验证。

## 13. FastAPI 和前端做了什么

FastAPI 提供了这些接口：

- `GET /`：Web 页面
- `GET /health`：健康检查
- `POST /api/chat`：问诊分诊接口
- `GET /api/metrics`：返回评测指标
- `POST /api/rebuild`：重建 SQLite 和 Chroma 向量库

Web 页面展示：

- 用户症状输入框
- 是否使用 LLM
- 是否使用 AgentExecutor
- 分诊答案
- 检索证据
- Agent 调用轨迹
- Top-1 / Top-3 评测指标

这部分主要体现工程落地能力，不只是一个命令行脚本。

## 14. 面试高频问题与回答

### 问题 1：你这个项目解决了什么问题？

回答：

它解决的是线上问诊中用户症状描述不规范、科室选择困难和医学回答缺少依据的问题。系统通过 RAG 检索医学知识库，再结合急症规则和 Agent 工具调用，给出可解释的分诊建议和初步健康提示。

### 问题 2：为什么不用大模型直接回答？

回答：

医疗场景不能完全依赖大模型直接生成，因为它可能幻觉、误判风险等级或给出不合适用药建议。我使用 RAG 把回答限制在检索到的医学知识范围内，同时核心分诊结果由规则和检索排序确定，LLM 只做补充说明。

### 问题 3：RAG 的召回质量怎么保证？

回答：

我用了混合检索。向量检索负责语义相似，关键词检索负责精确症状匹配。尤其医疗红旗词如胸痛、呼吸困难、血尿，必须通过关键词强化，避免只靠语义检索漏掉关键风险。

### 问题 4：为什么选择 Chroma？

回答：

项目规模较小，重点是本地可运行和 LangChain 集成。Chroma 支持本地持久化、元数据管理和快速接入，非常适合毕业设计 Demo。更大规模时可以替换为 Milvus、FAISS 或 Elasticsearch 混合检索。

### 问题 5：AgentExecutor 在项目中有什么意义？

回答：

AgentExecutor 把分诊流程拆成多个工具，包括急症筛查、医学检索、科室排序、追问和回答生成。它让模型不是一次性生成答案，而是按 ReAct 流程调用工具，过程可观察、可调试，也方便后续新增工具。

### 问题 6：如果模型调用工具不稳定怎么办？

回答：

我做了受控 ReAct 兜底。AgentExecutor 会先自主调用工具，如果没有完成关键步骤，比如没有调用检索或没有生成最终答案，系统会自动走固定工具链，保证输出稳定。

### 问题 7：Top-1 和 Top-3 指标怎么计算？

回答：

我构建了 20 条带标准科室标签的问诊样本。系统对每条样本输出 Top-3 科室，如果第一名等于标准科室，就算 Top-1 命中；如果标准科室在前三名中，就算 Top-3 命中。最后对所有样本求平均。

### 问题 8：为什么结果能达到 100%？

回答：

因为这是小规模自建评测集，覆盖的是知识库已经建模的常见场景，主要用于验证工程流程和算法闭环。它不能代表真实临床准确率。真实场景需要扩大样本、引入医生多标注、做交叉验证和错误分析。

### 问题 9：系统如何处理急症？

回答：

系统维护了一组急症红旗词，比如胸痛、呼吸困难、意识障碍、一侧无力、说话不清等。只要命中这些信号，系统会优先提示急诊或拨打急救电话，而不是给普通门诊建议。

### 问题 10：如何减少医疗幻觉？

回答：

主要有四个措施：

- 回答必须基于检索证据。
- 核心科室和风险等级由规则计算，不让 LLM 随意修改。
- 禁止输出具体处方剂量。
- 对危险措辞做安全替换，例如不建议“开具药物”，只提示遵医嘱或咨询医生药师。

### 问题 11：项目还有什么不足？

回答：

目前知识库和评测集规模仍然较小，医学知识是人工整理的样例，不具备真实临床上线能力。向量 embedding 使用的是 DeepSeek R1 本地接口，不是专门医学 embedding。后续可以扩充真实问诊数据、引入医学中文 embedding、增加医生标注评测，并加入多轮追问策略。

### 问题 12：如果继续优化，你会怎么做？

回答：

我会从四个方向优化：

- 数据层：扩充疾病、症状、药品、禁忌症和科室规则。
- 检索层：加入 BM25 + 向量检索 + rerank。
- Agent 层：让追问策略更细，比如根据不同科室生成不同问题。
- 评测层：构建更大规模标注集，统计各科室混淆矩阵和误诊风险样本。

## 15. 可以背的项目介绍版本

我做的是一个面向线上问诊的智能医疗分诊系统。用户输入自然语言症状后，系统会先进行急症红旗识别，然后通过 Chroma 向量数据库和关键词混合检索，从医学知识库中召回相关疾病、症状、药品和分诊规则。之后系统会根据检索相关性、风险等级和关键词命中情况，对候选科室进行 Top-3 排序。

在大模型部分，我通过 Ollama 本地部署了 `deepseek-r1:7b`，并用 LangChain 接入 LLM 和 AgentExecutor。AgentExecutor 中定义了急症筛查、医学检索、科室排序、追问和回答生成等工具，使模型能够按照 ReAct 流程调用工具，而不是直接生成不可控答案。

考虑到医疗场景对安全性要求较高，我没有让 LLM 直接决定最终分诊结果，而是让规则和检索算法先确定核心结论，LLM 只基于检索证据做补充说明。项目还实现了 FastAPI 接口和 Web 前端，并构建了 20 条标注问诊集进行评测，目前在该小规模评测集上 Top-1 分诊准确率和 Top-3 召回率均为 100%。
